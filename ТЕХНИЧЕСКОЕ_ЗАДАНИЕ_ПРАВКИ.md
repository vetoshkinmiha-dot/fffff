### 0.1 Перед любыми изменениями сделать бэкап PostgreSQL

## 1. АКТЫ НАРУШЕНИЙ — ПОЛНАЯ ПЕРЕРАБОТКА

### 1.1 Видимость актов нарушений по ролям

| Роль | Что видит во вкладке «Акты нарушений» |
|------|--------------------------------------|
| **Сотрудник подрядной организации** (contractor_employee) | Только нарушения, поданные на ЕГО организацию |
| **Ответственный подрядной организации** (contractor_admin) | Только нарушения, поданные на ЕГО организацию |
| **Сотрудник завода** (employee) | ТОЛЬКО те нарушения, которые ОН САМ подал. Не видит другие акты и жалобы подрядчиков |
| **Согласующий департамента** (department_approver) | ТОЛЬКО те нарушения, которые ОН САМ подал. Не видит другие акты и жалобы подрядчиков |
| **Администратор** (admin) | ВСЕ акты нарушений и ВСЕ жалобы. Имеет фильтр по типу: «Акты нарушений» / «Жалобы» |

### 1.2 Жалобы подрядчиков — полная переработка

#### 1.2.1 Жалоба становится свободной (не привязана к конкретному акту)
- Жалоба может быть подана БЕЗ привязки к конкретному акту нарушения
- Опционально может быть привязана к акту (поле `violationId` — nullable)
- Каждая жалоба привязана к `contractorId` (организация подрядчика)
- У жалобы есть автор (`createdById` — User)

#### 1.2.2 Кнопка «Подать жалобу»
- Отображается у ВСЕХ сотрудников подрядных организаций (contractor_employee И contractor_admin)
- Находится на ОСНОВНОЙ странице «Акты нарушений» (/violations)
- Располагается НА МЕСТЕ кнопки «Создать акт» (которая у администратора)
- НЕ зависит от конкретного акта нарушения
- Открывает модальное окно с формой:
  - Департамент (выпадающий список): HSE, Куратор договора, Закупки, Качество, Юридический, Финансовый, HR
  - Текст жалобы (textarea)
  - Опционально: привязка к акту нарушения (dropdown с номерами актов на их организацию)

#### 1.2.3 Удалить блок «Жалоба подрядчика» из карточки акта нарушения
- В `app/violations/[id]/page.tsx` убрать секцию с формой жалобы
- Жалобы теперь подаются через отдельную кнопку на списке

#### 1.2.4 Комментарий подрядчика в карточке акта
- В карточке акта нарушения (detail page) добавить поле «Комментарий подрядчика»
- Поле доступно для редактирования: contractor_admin и contractor_employee (только если акт относится к их организации)
- Поле read-only для admin/employee/department_approver
- Данные сохраняются в поле `contractorComment` модели Violation

#### 1.2.5 Администратор рассматривает жалобу
- Администратор может открыть карточку жалобы и принять решение:
  - Рассмотреть (resolved) — с примечаниями
  - Отклонить (rejected) — с примечаниями
- Примечания админа сохраняются в `resolutionNotes` жалобы

### 1.3 Отдельная страница жалоб у администратора
- У администратора на странице /violations должен быть переключатель-фильтр:
  - «Акты нарушений» (по умолчанию)
  - «Жалобы»
- При выборе «Жалобы» показывается таблица всех жалоб от всех подрядчиков:
  - Номер/ID жалобы
  - Подрядная организация
  - Департамент
  - Текст жалобы
  - Привязанный акт (если есть)
  - Статус (ожидает / рассмотрено / отклонено)
  - Кто подал (ФИО)
  - Дата подачи
  - Действия: открыть карточку, рассмотреть, отклонить

### 1.4 Миграция данных
- Existing ViolationComplaint records (привязанные к violationId) должны остаться и получить `status = pending`
- У existing жалоб `createdById` нужно заполнить (взять из связанного violation.createdBy или оставить null)
- У existing жалоб `contractorId` заполнить из связанного violation.contractorId

## 2. НАРЯДЫ-ДОПУСКИ — ЛОГИКА СТАТУСОВ

### 2.1 Убрать статус «Черновик» (draft)
- Удалить `draft` из enum `PermitStatus`
- Удалить все ссылки на `draft` в коде (frontend + backend)
- При создании наряда допуск сразу получает один из двух статусов:

### 2.2 Логика создания наряда
| Кто создаёт | Начальный статус | Что происходит дальше |
|-------------|-----------------|----------------------|
| **Администратор** | `active` (открытый) | Наряд сразу открыт, админ может его закрыть досрочно |
| **Ответственный подразделения** (department_approver) | `pending_approval` | Отправляется на согласование администратору |

### 2.3 Согласование наряда администратором
- Когда ответственный подразделения создаёт наряд → он попадает в «Согласования» администратора
- Администратор может:
  - **Согласовать** → статус наряда меняется на `active` (открытый)
  - **Отклонить** → статус наряда меняется на `closed` (закрыт)

### 2.4 Закрытие наряда
- **Администратор** может закрыть наряд досрочно (кнопка «Закрыть досрочно»)
- **Автоматическое закрытие** по истечении срока действия (`expiryDate` < today) — cron job

### 2.5 Видимость нарядов
| Роль | Что видит |
|------|-----------|
| **Сотрудник подрядной организации** | Все наряды, оформленные на ЕГО организацию (любые статусы) |
| **Ответственный подрядной организации** | Все наряды, оформленные на ЕГО организацию (любые статусы) |
| **Администратор** | Все наряды всех организаций |
| **Согласующий департамента** | Все наряды (просмотр) |
| **Сотрудник завода** (employee) | Все наряды (просмотр) |

## 3. СОГЛАСОВАНИЯ — ФИЛЬТР У АДМИНИСТРАТОРА

### 3.1 Во вкладке «Согласования» у администратора должен быть переключатель:

**Вариант A: «Согласования нарядов-допусков»**
- Список нарядов в статусе `pending_approval`
- Кнопки: «Согласовать» / «Отклонить»
- Для каждого наряда: номер, подрядчик, участок, ответственный, дата

**Вариант B: «Согласования сотрудников»**
- Подразделено по департаментам (табы):
  - Служба безопасности (security)
  - Отдел кадров (hr)
  - Охрана труда — допуск (safety)
  - Охрана труда — инструктаж (safety_training)
  - Бюро пропусков (permit_bureau)
- В каждом табе — список сотрудников на согласование
- Кнопки: «Согласовать» / «Отклонить»
- Для каждого сотрудника: ФИО, организация, должность, классы работ

### 3.2 У обычных согласующих (department_approver)
- Они видят ТОЛЬКО свой департамент (как сейчас)
- Без переключателя — только свои согласования

## 4. СОТРУДНИКИ — АВТО-СОЗДАНИЕ УЧЁТНОЙ ЗАПИСИ

### 4.1 При добавлении сотрудника (admin или contractor_admin)
- Автоматически создаётся User с ролью `contractor_employee`
- **Логин** = email, сформированный автоматически:
  - Формат: `{транслит_ФИО}@{домен_организации}.ru`
  - Или: `employee_{random}@vshz.ru` (если домен организации не задан)
- **Пароль** = случайная строка из 12 символов (буквы + цифры)
- User привязан к организации сотрудника (`organizationId`)
- User привязан к сотруднику через `employeeId`
- `mustChangePwd = true` (при первом входе сменит пароль)

### 4.2 Уведомление о создании учётки
- После успешного добавления сотрудника появляется плашка:
  - «Сотрудник [ФИО] успешно добавлен»
  - Кнопка «Скачать логин и пароль» (.txt файл)
- Файл содержит:
  ```
  Сотрудник: [ФИО]
  Логин: [email]
  Пароль: [password]
  ```

### 4.3 Учётки на экране логина
- На странице логина (/login) под формой входа отображается список всех учётных записей `contractor_employee`
- Для каждой: ФИО, логин (email), пароль (открытым текстом)
- Только для удобства разработки — в production это убрать
- Сортировка: сначала новые

### 4.4 При удалении сотрудника (только admin)
- Автоматически удаляется связанная User запись (`contractor_employee`)
- Удаляются все EmployeeDocument
- Удаляются все ApprovalRequest
- Удаляется сам Employee
- CASCADE через Prisma relations

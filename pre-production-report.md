## КРИТИЧЕСКИЕ (должны быть исправлены до прода)

### 1. JWT Secret fallback
**Файл:** lib/auth.ts:6, lib/auth-edge.ts:5
**Проблема:** Если `JWT_SECRET` не установлен, используется `"dev-secret-change-in-production"`. Любой знает этот секрет и может подделать админский токен.
**Решение:** Бросать ошибку при старте если секрет не задан или равен дефолтному.

### 2. Path traversal в удалении документов
**Файл:** app/api/documents/regulatory/[id]/route.ts (DELETE handler, строка ~84)
**Проблема:** `filePath = path.join(process.cwd(), doc.fileUrl)` — без проверки что путь внутри uploads/. Если БД скомпрометирована, можно удалить любой файл.
**Решение:** Добавить `startsWith(uploadsDir)` проверку.

### 3. Path traversal в photo API
**Файл:** app/api/employees/[id]/photo/route.ts (строки ~66, ~101)
**Проблема:** `getFile()` и `deleteFile()` используют путь из БД без санитизации.
**Решение:** Санитизировать пути в lib/file-storage.ts.

### 4. Employee не может редактировать/удалять СВОИ нарушения
**Файл:** app/api/violations/[id]/route.ts (PATCH, DELETE)
**ТЗ п.3.2:** Сотрудник завода должен иметь возможность заполнять и выгружать свои акты нарушений.
**Проблема:** Только admin/department_approver могут редактировать. Employee не может.
**Решение:** Разрешить PATCH/DELETE создателю нарушения (если role === employee).

### 5. Нет CRUD для жалоб
**Файл:** app/api/violations/[id]/complaints/route.ts
**ТЗ п.3.3:** «удаление/редактирование только своих жалоб»
**Проблема:** Только POST есть. Нет PATCH/DELETE/GET для жалоб.
**Решение:** Добавить endpoint'ы GET/PATCH/DELETE для complaints.

## ВЫСОКИЙ ПРИОРИТЕТ

### 6. CSRF защита недостаточная
**Файл:** lib/api-middleware.ts
**Проблема:** Проверка Content-Type легко обходится. SameSite=Strict — хорошо, но не покрывает XSS на том же домене.
**Решение:** Добавить CSRF token в JWT payload и проверять заголовок X-CSRF-Token для мутаций.

### 7. Нет печати наряд-допусков (ТЗ п.2.2)
**Файл:** app/permits/[id]/print/page.tsx
**Проблема:** Есть страница /print но она просто показывает HTML. Нет форматированной печати с печатями/подписями.
**Решение:** Доработать print страницу с CSS @media print, правильным форматированием.

### 8. Нет печати актов нарушений (ТЗ п.3.2)
**Файл:** app/violations/[id]/print/page.tsx
**Проблема:** Аналогично — HTML без форматированного макета для печати.
**Решение:** Доработать print страницу.

### 9. contractor_employee может редактировать карточку любого сотрудника в org через список
**Файл:** app/employees/[id]/route.ts (PATCH)
**ТЗ:** «просмотра и редактирования только своей карточки»
**Проблема:** PATCH проверяет org но не проверяет employeeId.
**Решение:** Добавить `authResult.user.employeeId === id` проверку для contractor_employee.

### 10. MustChangePwd не enforced серверно
**Файл:** middleware.ts, app/api/auth/me
**Проблема:** Флаг `mustChangePwd` есть, но middleware не перенаправляет на /change-password при mustChangePwd=true.
**Решение:** Добавить проверку в middleware.

## СРЕДНИЙ ПРИОРИТЕТ

### 11. Race condition в sequential number организаций
**Файл:** app/api/organizations/route.ts (POST)
**Проблема:** aggregate MAX без Serializable транзакции.
**Решение:** Обернуть в $transaction с Serializable.

### 12. Отсутствие cascade delete
**Файл:** prisma/schema.prisma
**Проблема:** При удалении Organization: Permit, Violation, Checklist, User — не каскадируются. Оставшиеся записи становятся orphaned.
**Решение:** Добавить `onDelete: Cascade` или `onDelete: SetNull` на соответствующих полях.

### 13. Permit status = "closed" при отклонении
**Файл:** app/api/permits/[id]/approvals/[approvalId]/route.ts
**Проблема:** При отклонении permit получает status "closed" — но это путает с "закрыт нормально". Должен быть "rejected" или "draft".
**Решение:** Использовать отдельный статус для отклонённых нарядов.

### 14. Нет file size limit на regulatory docs
**Файл:** app/api/documents/regulatory/route.ts (POST)
**Проблема:** Нет проверки размера файла при загрузке.
**Решение:** Добавить проверку file.size < 50MB (или другой лимит).

### 15. Employee не может видеть свои нарушения в списке
**Файл:** app/api/violations/route.ts (GET)
**ТЗ:** Сотрудник завода видит «свои поданные акты нарушений»
**Проблема:** Фильтр по createdById работает, но нет визуальной метки «мои нарушения» на фронте.
**Решение:** Добавить на фронт фильтр «Мои нарушения».

## НИЗКИЙ ПРИОРИТЕТ (UX/UI улучшения)

### 16. Пустые строки в SelectValue
**Файл:** app/documents/page.tsx (SelectValue children functions)
**Проблема:** При пустом значении placeholder не показывается — `"" ?? "placeholder"` возвращает `""`.
**Решение:** Использовать `v ? ... : placeholder`.

### 17. Нет лоадера при загрузке страницы нарушений
**Файл:** app/violations/page.tsx
**Проблема:** Нет индикации загрузки.
**Решение:** Добавить Skeleton/Loader.

### 18. Дублирование текста в sidebar
**Файл:** components/layout/sidebar.tsx
**Проблема:** ROLE_VISIBLE_NAV дублирует логику из middleware.ts. Если меняется middleware — нужно менять sidebar вручную.
**Решение:** Вынести конфигурацию в отдельный файл (lib/roles.ts).

### 19. Нет индикатора «Мои нарушения» в списке
**Файл:** app/violations/page.tsx
**Проблема:** Пользователь не видит какие нарушения — его.
**Решение:** Добавить бейдж «Моё» к своим нарушениям.

### 20. Таблица permit без мобильной адаптации
**Файл:** app/permits/page.tsx
**Проблема:** Нет mobile cards как на checklists.
**Решение:** Добавить mobile view.
